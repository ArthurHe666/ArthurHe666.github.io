浅谈paxos算法

背景

​ 在分布式系统中，无法同时满足数据高一致性和高可用，所以为了满足在节点异常的场景下，数据读写的一致性，提出了数据一致性算法paxos

定义

​ paxos算法需要解决的问题就是分布式系统如何就某个值达成一致

特点

​ paxos算法是一种投票算法，在允许宕机故障的分布式系统中，它利用投票机制保证了2F+1个节点的容错能力，即2F+1个节点的系统最多允许F个节点同时出现故障

​ paxos没有解决同时满足数据高一致性和高可用这个问题，只是在数据高一致性和高可用之间取得了更好的平衡，即故障场景下，从内部看，某些节点的数据可能是落后的，但是上层应用可以访问到强一致性的数据

原理

Paxos将系统中的角色分为提议者 (Proposer)，决策者 (Acceptor)，和最终决策学习者 (Learner)

Proposer ： 发起提案(proposal)

Acceptor ： 对提案进行投票(同意/拒绝)

Learner ： 不参与提案流程，同步达成一致的提案 (Proposer)

proposal ： 提案，由提案编号N和提案内容value组成(N， value)

Paxos算法通过一个提案分为两个阶段，如图所示：

prepare阶段

Proposer : Proposer发起提案proposal，proposal中只用携带提案编号N，向所有的Acceptor广播prepare请求。

Acceptor ： 接收提案proposal，进行promise流程；

​ 如果当前prepare请求的提案编号N大于Acceptor之前接收到的所有的提案编号minProposal，就接收该提案，minProposal = N，并承诺不在接收比提案编号N小的提案，如果存在已经同意的提案(acceptedProposal, acceptedValue)，则返回(acceptedProposal, acceptedValue)给Proposer ，如果不存在已经同意的提案，则返回(NULL, NULL)给Proposer

​ 如果当前的prepare请求的提案编号N小于已经收到的提案minProposal，则拒绝，返回当前接收的提案(acceptedProposal, acceptedValue)给Proposer，如果不存在已经同意的提案，则返回(NULL, NULL)给Proposer

Proposer ： 接收到大多数的Acceptor的回复后，如果发现有acceptedValue返回，将所有回复的(acceptedProposal, acceptedValue)中最大的acceptedProposal对应的acceptedValue作为本次提案的value，如果acceptedValue为空，则可以任意决定本次提案的value；

accept阶段

​ Proposer : 广播(N, value)给所有Acceptor节点

​ Acceptor ： 比较N和当前的minProposal，如果N >= minProposal，则更新acceptedValue = value，同时更新minProposal = N，返回 minProposal；如果N < minProposal，返回 minProposal

​ Proposer : 接收到大多数的Acceptor的回复后，如果发现minProposal大于当前提案编号N，提案有更新，提案失败，需要重新开始提案，否则，value达成一致，提案通过

以下通过两个例子来更具体描述paxos的提案流程

实例1

Proposer_1发起提案，提案编号为n

Acceptor_1、Acceptor_2、Acceptor_3接收提案，之前没有接收过提案，更新minProposal为n

Acceptor_1、Acceptor_2、Acceptor_3返回(null, null)

Proposer_1接收到了大多数Acceptor的回复，查询是否存在acceptedValue，不存在则自己创建本次提案value

Proposer_1发送accept请求(n, value)

Acceptor_1、Acceptor_2、Acceptor_3接收accept请求，检查n >= minProposal，更新minProposal = n，更新

acceptedValue = value，返回minProposal

Proposer_1接收到大多数的Acceptor回复后，检查minProposal <= n，提案通过

实例2

Proposer_1发起提案，提案编号为n

Acceptor_1、Acceptor_2接收提案，之前没有接收过提案，更新minProposal为n

Acceptor_1、Acceptor_2返回(null, null)

Proposer_1接收到了大多数Acceptor的回复，查询是否存在acceptedValue，不存在则自己创建本次提案value1

​

Proposer_2发起提案，提案编号为n+1

Acceptor_2判断minProposal < n+1，更新minProposal=n+1, 并保证不会接收小于n+1的提案

Acceptor_3接收提案，之前没有接收过提案，更新minProposal为n

Acceptor_3、Acceptor_2返回(null, null)

Proposer_1接收到了大多数Acceptor的回复，查询是否存在acceptedValue，不存在则自己创建本次提案value2

Proposer_1发送accept请求(n, value1)

Acceptor_1接收accept请求，检查n >= minProposal，更新minProposal = n，更新 acceptedValue = value1，返回minProposal

Acceptor_2接收accept请求，检查n < minProposal，拒绝本次提案，返回minProposal

Proposer_1接收到大多数的Acceptor回复后，检查minProposal > n，提案失败，提案有更新，重新提案

Proposer_2发送accept请求(n+1, value2)

Acceptor_2接收accept请求，检查n >= minProposal，更新minProposal = n+1，更新 acceptedValue = value2，返回minProposal

Acceptor_3接收accept请求，检查n >= minProposal，更新minProposal = n+1，更新 acceptedValue = value2，返回minProposal

Proposer_2接收到大多数的Acceptor回复后，检查n >= minProposal，提案通过

Proposer_1重新发起提案，提案编号为n+2

Acceptor_1判断minProposal < n+2，更新minProposal=n+2, 并保证不会接收小于n+1的提案，同时返回(n, value1)

Acceptor_2判断minProposal < n+2，更新minProposal=n+2, 并保证不会接收小于n+1的提案，同时返回已经通过的提案(n+1, value2)

Proposer_1接收到了大多数Acceptor的回复，查询存在acceptedValue，且n+1的提案较新，更新本次提案的value为value2

Proposer_1发送accept请求(n+2, value2)

Acceptor_2接收accept请求，检查n >= minProposal，更新minProposal = n+2，更新 acceptedValue = value2，返回minProposal

Acceptor_3接收accept请求，检查n >= minProposal，更新minProposal = n+2，更新 acceptedValue = value2，返回minProposal

Proposer_1接收到大多数的Acceptor回复后，检查n >= minProposal，提案通过

缺点

paxos算法有如下缺点

1）多个Proposer进行prepare成功，但是因accept请求失败，从而继续重新开始提案，导致集群永远无法达成一致，出现死循环，也就是所谓的活锁

2） 每个提案至少需要经过两轮(prepare和accept)，如果有多个提案者，可能导致提案通过效率大大降低

改进方案

1）针对活锁问题，提案失败后，进行一定时间的等待，等待集群完成一个提案

2） 针对提案效率低的问题，使用更强大的raft和multi-paxos算法
